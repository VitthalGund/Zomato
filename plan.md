# ENGINEERING SPECIFICATION: ZOMATO EDGEVISION

## 1. System Objectives & Constraints

* **Primary Objective:** Replace manual, heavily biased "Food Ready" (FOR) merchant signals with an automated, cryptographically secure timestamp generated by an Edge AI model.
* **Secondary Objective:** Use the velocity of these verified timestamps to mathematically infer the "invisible" non-Zomato kitchen load (dine-in, Swiggy, etc.) to dynamically recalibrate Zomato's Kitchen Prep Time (KPT).
* **Strict Constraints:**
* **Zero Cloud Video Streaming:** Indian kitchen internet is terrible. The camera frame **must not** leave the device.
* **Sub-200ms Latency:** The system cannot block the merchant's operational flow.
* **Zero New Hardware:** Must run entirely within the existing Zomato Mx Android/iOS application.

## 3. Edge Client Architecture (Zomato Mx App Integration)

This is the hardest part of the engineering. The mobile app must capture frames, run inference, and generate a secure payload without melting the phone's battery.

### 3.1. Camera Pipeline & Throttling

* **Do NOT run inference at 30 FPS.** * **Implementation:** The camera buffer must throttle frame capture to **3 FPS** (Frames Per Second). Downscale the image resolution to `640x640` before passing it to the inference engine.

### 3.2. Local Inference Logic (The Bounding Box Intersect)

The object detection model looks for two classes: `Class 0: Food_Parcel` and `Class 1: KOT_Receipt`.

* **The Logic Gate:** The system must detect *both* classes in the same frame.
* **Spatial Requirement:** The bounding box of `Class 1` MUST intersect or be fully contained within the bounding box of `Class 0`. A receipt sitting on a table next to a bag is a failure state.
* **State Debouncer:** To prevent motion-blur false positives, the logic gate must return `TRUE` for **3 consecutive frames** before triggering a successful verification.

### 3.3. Cryptographic Payload Generation

Once verified, the client does not send the image. It generates a secure JSON payload.

* Pull the active `Order_ID` from the app's UI state.
* Generate a local Unix timestamp.
* Generate a SHA-256 Hash: `Hash(Order_ID + Timestamp + Merchant_Device_ID + Secret_Salt)`.
* **Output Payload:**

```json
{
  "order_id": "ZOM-98765",
  "verified_timestamp": 1709368400,
  "confidence_score": 0.94,
  "crypto_hash": "a8f5f167f44f4964e6c998dee827110c",
  "fallback_used": false
}

```

### 3.4. Fallback Mechanisms (Critical for UI/UX)

* **T+5 Seconds (Lighting Protocol):** If inference fails for 5 continuous seconds, the app automatically triggers the device hardware flashlight API.
* **T+10 Seconds (Manual Override):** If inference fails for 10 seconds, render a "Manual Override" button. If clicked, the app requires the merchant to manually type the last 4 digits of the Order ID. The JSON payload changes `fallback_used` to `true`.

## 4. Cloud Backend Architecture (FastAPI & Data Processing)

The backend has two jobs: ingest the high-throughput JSON payloads and sanitize them for fraud.

### 4.1. The Ingestion Microservice (FastAPI)

* Expose an asynchronous POST endpoint: `/api/v1/verify-order`.
* **Validation:** Verify the `crypto_hash` to ensure the payload was generated by the authorized Zomato app and not a spoofed API request.

### 4.2. Signal Sanitizer (Fraud Detection)

Merchants will try to scan empty bags immediately after receiving an order to cheat the metrics. You must build a temporal validation gate.

* **Logic:** Calculate `Delta = verified_timestamp - order_creation_timestamp`.
* **Check:** Query the database for the `minimum_prep_time` of the specific items in that order (e.g., a raw chicken biryani cannot physically be cooked in 3 minutes).
* **Action:** If `Delta < minimum_prep_time`, flag the payload as `FRAUD_EMPTY_BAG`. Store it in the database but **do not** forward it to the KPT model.

## 5. Core Processing Engine: Throughput Velocity (Little's Law)

This is the algorithm that wins you the hackathon. This calculates the non-Zomato kitchen rush without needing API access to Swiggy or the POS system.

### The Algorithm

Implement a background worker (e.g., Celery or a background FastAPI task) that runs every 5 minutes for every active merchant.

1. **Calculate Baseline Velocity ($V_b$):** The historical average number of verified orders completed by this merchant between 7 PM and 8 PM on a Friday (e.g., 20 orders/hour).
2. **Calculate Current Velocity ($V_c$):** The moving average of verified Zomato orders completed in the last 15 minutes, extrapolated to an hourly rate (e.g., 5 orders/hour).
3. **Calculate Invisible Load Ratio ($R_i$):** 
$$R_i = \frac{V_c}{V_b}$$


4. **Execution Logic:** * If $R_i \approx 1$, the kitchen is operating normally. Proceed with standard ETA.
* If $R_i < 0.5$ (meaning output has dropped by 50% despite active Zomato orders pending), the backend mathematically infers the kitchen is overwhelmed by invisible walk-in/competitor traffic.
* **Action:** The system automatically adds a dynamic penalty (e.g., +12 minutes) to the KPT prediction, intentionally delaying the dispatch of the next delivery rider.
